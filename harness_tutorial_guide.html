calc-harness 하네스 실습 가이드

Claude Code로 훅(hook) · 스킬(skill) · 에이전트(agent)를 하나씩 직접 만들며 익힌 전 과정 기록이다. 간단한 Rust 계산기(calc-harness)를 소재로, 0단계 셋업부터 6단계 에이전트까지 만들면서 실제로 겪은 오류와 그 해결 과정까지 그대로 담았다.

이 문서의 클로드 코드 개념·문법은 전부 공식 문서(code.claude.com)에서 확인한 내용만 담았고, 근거 페이지는 각 절과 맨 끝 "공식 문서 출처"에 링크로 남겼다. 실습 기록은 실제로 일어난 그대로이며, 잘못 짚었던 진단까지 포함한다.

작업 환경은 Windows + cmd(명령 프롬프트), 훅 스크립트는 PowerShell로 작성했다.

1. 하네스란 무엇인가

하네스(harness)는 원래 말·낙하산에 채우는 마구·안전벨트 장치를 뜻한다. Claude Code에서는 AI를 원하는 방향으로 몰기 위해 주변에 둘러치는 장치 전체, 즉 프로젝트의 .claude/ 폴더 전체가 곧 하네스다.

하네스가 필요한 이유는 AI에 두 가지 근본 한계가 있기 때문이다.

  • 작업 기억(컨텍스트)이 유한하다 — 소스를 잔뜩 읽으면 그 덤프로 기억이 차서 중요한 맥락을 잊는다.
  • 지시를 확률적으로 따른다 — "이거 하지 마"라고 해도 가끔 어긴다. 지시는 부탁이지 보장이 아니다.

세 가지 도구는 각각 이 한계를 보완한다.

세 요소 요약

정의 발동 성격
에이전트 별도 컨텍스트에서 도는 전문 분석가. 결론만 회수 위임(능동) 컨텍스트 절약
스킬 특정 작업의 표준 절차서·규약 부르거나 자동 로드(능동) 품질·일관성
훅 특정 이벤트에 자동 실행되는 스크립트 이벤트 발생 시(자동) 안전장치·자동화

결정론 vs 확률론 — 문서를 관통하는 축

핵심은 "반복이냐"가 아니라 "누가 언제 발동시키느냐"다. 공식 문서도 이 구분을 명시한다: 훅은 결정론적(정해진 시점에 예외 없이 실행)이고, 스킬·에이전트는 확률적(Claude가 판단해 적용)이다.

반드시·매번·예외 없이 지켜야 하면 → 훅. 판단에 맡겨도 되면 → 스킬·에이전트.

한 요청 안에서 셋이 어떻게 맞물리는지 보면 전체 그림이 잡힌다.

2. 훅 (Hook)

훅은 Claude Code 생명주기의 특정 지점에서 자동 실행되는 명령이다. 프롬프트가 아니라 결정론적 코드라, AI가 건너뛸 수 없다. (공식 훅 레퍼런스 · 가이드)

훅이 세션의 어느 지점에서 발동하는지는 공식 문서의 생명주기 그림으로 한눈에 볼 수 있다(Setup → SessionStart → … → SessionEnd).

같은 흐름을 세 리듬(세션 · 턴 · 도구 루프)으로 요약하면 이렇다. 도구를 쓸 때는 항상 PreToolUse에서 시작해 도구 실행 → PostToolUse → Claude가 결과를 판단하고, 도구가 더 필요하면 반복한다. 정상 완료면 Stop, API 에러로 끝나면 StopFailure다.

가장 중요한 구분은 실행 전(Pre) 이냐 실행 후(Post) 냐다.

  • PreToolUse — 도구 실행 직전. exit 2로 차단할 수 있다.
  • PostToolUse — 도구 성공 후. 이미 실행됐으므로 차단은 못 하고, 뒷정리와 피드백만 한다.

훅 이벤트 종류

공식 문서 기준 훅 이벤트는 약 30종이며, 언제 발동하느냐(cadence)로 묶으면 이해가 쉽다.

  • 세션당 1회: SessionStart, SessionEnd
  • 턴당 1회: UserPromptSubmit, Stop, StopFailure
  • 도구 호출마다: PreToolUse, PostToolUse

실습에서 실제로 쓰는 것만 추리면 이 정도다.

이벤트 발동 시점 차단 가능 쓰임새
PreToolUse 도구 실행 직전 가능(exit 2) 위험 명령·값 차단
PostToolUse 도구 성공 후 불가 포맷·정리
SessionStart 세션 시작 불가 문맥 주입
UserPromptSubmit 프롬프트 제출 가능 프롬프트 검증
Stop 턴 종료 가능 종료 후 테스트
SubagentStop 서브에이전트 종료 가능 후처리

훅 설정 위치와 형식

훅은 JSON 설정 파일에 정의한다. 위치에 따라 범위가 다르다: .claude/settings.json(프로젝트, 공유 가능), ~/.claude/settings.json(사용자 전역), 그리고 스킬·에이전트 frontmatter나 플러그인에도 넣을 수 있다.

구조는 이벤트 → matcher 그룹 → 핸들러 3단계다.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "...", "timeout": 10 }
        ]
      }
    ]
  }
}
  • matcher — 어떤 도구/이벤트에 걸지. "Bash"(정확히 일치), "Edit|Write"(둘 중 하나), "*" 또는 ""(전체). 도구 이벤트에서는 도구 이름과 매칭한다.
  • exit 코드 — exit 0은 성공(진행), exit 2는 차단(stderr가 Claude에게 에러로 전달), 그 외는 비차단 에러(동작은 진행). 보안 훅은 반드시 exit 2를 써야 실제로 막힌다.
  • 훅은 이벤트 발생 시 JSON을 stdin으로 받는다. 도구 이벤트면 tool_name, tool_input(예: .command, .file_path, .new_string)이 들어 있다.

/hooks를 치면 등록된 훅을 읽기 전용으로 확인할 수 있다(개수·matcher·명령).

Windows에서 훅

훅 핸들러는 기본적으로 셸 명령이다. Windows에서 JSON을 다루려면 PowerShell이 편하다. 훅 항목에 "shell": "powershell"을 주거나, command에서 powershell ... -File ...로 직접 호출하면 된다(항상 설치돼 있음). 이 실습에서는 후자를 썼다.

경로는 $CLAUDE_PROJECT_DIR(프로젝트 루트, Claude Code가 자동 주입)을 쓰면 작업 디렉토리와 무관하게 안정적이다.

3. 스킬 (Skill)

스킬은 특정 작업의 표준 절차·규약을 담은 문서다. 필요할 때 그 규약이 작업 지침으로 로드된다. (공식 스킬 문서)

스킬은 .claude/skills/<이름>/SKILL.md 구조이며, 두 부분으로 이뤄진다: YAML frontmatter(언제 쓸지)와 마크다운 본문(어떻게 할지).

---
name: add-error-code
description: 새 에러코드를 추가할 때 사용. "에러 추가", "errorCode 추가"를 요청하면 따른다.
---

# 규약 본문 ...
  • 필수 필드는 name(폴더명과 일치, 소문자·하이픈)과 description(무엇을 하고 언제 쓰는지 명시)뿐이다. description이 트리거를 좌우한다.
  • 호출은 두 가지다. /skill-name으로 직접 부르거나, Claude가 description이 요청과 맞는다고 판단하면 자동으로 편다. 즉 확률적이다.

점진적 공개 — 스킬이 가벼운 이유

평소에는 각 스킬의 이름과 description만(약 50토큰) 로드돼 있다가, 트리거될 때 본문 전체를 읽어 온다. 그래서 스킬을 수십 개 둬도 부담이 적다.

disable-model-invocation: true(자동 발동 끄고 수동만), user-invocable: false(사용자 슬래시 명령 끄고 Claude만) 같은 옵션으로 호출 방식을 제한할 수 있다.

4. 에이전트 (서브에이전트)

에이전트는 별도 컨텍스트 창에서 도는 독립 일꾼이다. 지저분한 작업(테스트 실행, 대량 파일 읽기)을 자기 방에서 하고, 메인 대화에는 결론(요약)만 돌려준다. (공식 서브에이전트 문서)

에이전트는 .claude/agents/<이름>.md 구조이며, frontmatter + 마크다운 본문(= 시스템 프롬프트)으로 이뤄진다.

---
name: test-analyst
description: 테스트를 실행하고 결과를 분석한다. "테스트 돌려줘"를 요청하면 사용한다.
tools: Bash, Read, Grep, Glob
model: inherit
---

너는 테스트 분석가다. cargo test를 돌리고 결론만 보고한다 ...
  • 필수는 name, description. tools를 생략하면 모든 도구를 상속하고, 지정하면 그것만 쓴다(예: Edit/Write를 빼면 읽기 전용 분석가). model은 기본 inherit.
  • 호출은 자연어("test-analyst 에이전트로 …"), @agent-<이름>, 또는 세션 전체를 그 에이전트로 여는 --agent 플래그가 있다.
  • 에이전트는 fresh 컨텍스트로 시작한다. 메인 대화 히스토리를 보지 않고, Claude가 넘긴 위임 메시지에서 출발한다.

5. 훅 vs 스킬 vs 에이전트

세 요소를 나란히 놓으면 차이가 분명해진다.

훅 스킬 에이전트
발동 이벤트에 자동 요청에 맞으면 펴짐 위임하면 뜸
어디서 도나 메인 대화 안 메인 대화 안 별도 컨텍스트
성격 결정론적 실행 규칙 안내 독립 분석·작업
돌려주는 것 차단 / 정리 방법 지침 결론(요약)
비유 안전벨트 매뉴얼 전문 직원

스킬에서 훅으로 — 이중 방어

스킬은 "부탁", 훅은 "강제"다. 스킬은 확률적이라 Claude가 안 펴거나 우회하면 뚫린다. 그래서 진짜 지켜야 하는 규칙은 스킬로 안내하고 훅으로 강제한다. 이 실습에서는 add-error-code 스킬이 "9xxx 쓰지 마"라고 말리고, guard-error-codes 훅이 9xxx 편집을 실제로 차단한다.

6. 프로젝트 구성 (calc-harness)

소재는 Rust CLI 계산기다. calc 10 / 2 처럼 숫자·연산자·숫자를 받아 계산하고, 오류는 에러코드로 표현한다.

최종 .claude/ 구조

calc-harness/
├── src/main.rs                        계산기 (에러코드 체계 + die() 헬퍼)
├── tests/cli.rs                       통합 테스트
└── .claude/
    ├── settings.json                  훅 3개 등록
    ├── hooks/
    │   ├── delete-guard.ps1           위험 삭제 차단 (PreToolUse)
    │   ├── guard-error-codes.ps1      9xxx 예약범위 차단 (PreToolUse)
    │   └── auto-fmt.ps1               편집 후 cargo fmt (PostToolUse)
    ├── skills/
    │   └── add-error-code/SKILL.md    에러코드 추가 규약
    └── agents/
        └── test-analyst.md            테스트 분석 에이전트

훅 3 + 스킬 1 + 에이전트 1. 실제 업무 프로젝트(에이전트 5 + 스킬 4 + 훅 4)의 축소판이다.

errorCode 번호 범위

에러를 범주별 번호대로 관리한다. 새 코드는 성격에 맞는 범위의 다음 빈 번호로 추가한다(스킬 규약).

7. 단계별 실습 (0~6)

각 단계는 만들고 → 테스트하고 → 다음으로 진행했다.

0단계 — 셋업

cargo new calc-harness --bin
cd calc-harness
git init && git add -A && git commit -m "init"

.claude 폴더와 빈 설정 파일을 만든다. Windows에서는 mkdir -p가 안 되고, echo '{}' > file은 따옴표까지 들어가거나 BOM이 붙을 수 있어 주의한다(아래 "Windows/cmd 팁" 참고). 테스트: cargo run으로 기본 실행 확인.

git 원격에 올리기

gh CLI 없이 웹에서 빈 저장소를 만들고 연결한다.

git remote add origin https://github.com/<계정>/calc-harness.git
git branch -M main
git push -u origin main
  • 저장소를 만들 때 README·.gitignore·license는 체크 해제해야 충돌 없이 push된다.
  • 첫 push 때 브라우저에 Git Credential Manager 인증 창이 뜨면 로그인하면 된다. 터미널에서 비밀번호를 물으면 계정 비번이 아니라 Personal Access Token이 필요하다.
  • Repository not found 에러는 대개 웹에 저장소가 아직 안 만들어진 것이다.

Windows/cmd 주의점

  • mkdir -p → PowerShell/cmd에서는 -p 불필요. cmd는 mkdir .claude\hooks.
  • cmd에서 %는 특수문자라 인자로 넘길 때 %%로 이스케이프.
  • del/rmdir은 delete-guard 패턴에 걸리므로, 진짜 지울 땐 사람이 직접 cmd에서 실행.

1단계 — 최소 계산기

코드를 손으로 치지 말고 Claude Code에게 시킨다. 그래야 이후 훅이 "편집하는 순간"에 걸리는 것을 볼 수 있다. + - * /를 지원하는 계산기를 만들게 하고, cargo run -- 10 / 2 → 5로 확인한다.

2단계 — delete-guard (PreToolUse 차단)

첫 훅. Claude Code가 rm 같은 삭제 명령을 실행하려 하면 차단한다.

자연어 요청이 어떻게 실제 명령이 되고, 훅이 어디서 끼어드는지 보자.

훅이 검사하는 것은 네 한국어 문장이 아니라 Claude가 번역해 만든 실제 명령어 문자열이다. 그래서 어떻게 말하든("지워줘", "삭제해") 결국 rm/Remove-Item 같은 명령이 만들어지고 패턴에 걸린다.

핵심 흐름을 단순화하면 이렇다.

스크립트(delete-guard.ps1, 부록 참고)는 stdin JSON에서 명령을 뽑아 위험 패턴과 대조하고, 걸리면 exit 2로 차단한다. 설정은 PreToolUse + matcher: "*"(모든 도구)로 걸었다. 왜 "*"인지는 8장의 실패 스토리에서 설명한다.

3단계 — auto-fmt (PostToolUse 자동 정리)

성격이 정반대인 훅. .rs 파일이 편집되면 편집 후에 cargo fmt를 자동 실행한다. 차단(exit 2)을 쓰지 않고, 그냥 정리하고 조용히 exit 0. 일부러 들여쓰기를 엉망으로 넣어도 저장 시점에 반듯하게 정돈된다.

if ($path -notmatch '\.rs$') { exit 0 }   # .rs 아니면 통과
cargo fmt 2>&1 | Out-Null
exit 0

matcher: "Edit|Write" — 파일을 편집/작성할 때만 건다.

4단계 — add-error-code (스킬)

먼저 계산기에 에러코드 체계(1xxx/2xxx/9xxx)를 심고, 그 코드를 규칙대로 추가하는 스킬을 만든다.

스킬을 만든 뒤 "오버플로우 에러 추가해줘"라고 시키면, Claude가 스킬을 펴서 스스로 판단한다: 오버플로우는 수학 오류(2xxx)이고 2001이 이미 있으니 다음 빈 번호 2002를 골라 [E2002] 형식으로 추가한다. 번호를 알려주지 않았는데 규약을 따라 정한 것이다 — 이것이 스킬이다.

훅과의 대비: 같은 편집에서 auto-fmt 훅도 자동으로 돈다. 스킬은 규칙을 따르고(판단 개입), 훅은 서식을 무조건 정리(판단 없음)한다.

5단계 — guard-error-codes (스킬 → 훅 이중 방어)

add-error-code 스킬은 "9xxx 쓰지 마"라고 말하지만 그건 부탁이다. 진짜 막으려면 훅이 필요하다. .rs 편집 내용에 9xxx가 들어가려 하면 exit 2로 차단한다.

여기서 순서가 중요하다: 스킬이 먼저, 훅이 나중이다. 스킬은 Claude가 "무엇을 할지 생각하는 단계"에 작동하고(그래서 착하면 편집을 시도조차 안 함), 훅은 "실제로 도구를 실행하려는 순간"에 작동한다. 그래서 훅을 진짜 테스트하려면 스킬을 우회시켜 편집을 유도해야 한다.

6단계 — test-analyst (에이전트)

먼저 tests/cli.rs에 통합 테스트를 넣고, cargo test를 별도 컨텍스트에서 돌려 결론만 회수하는 읽기 전용 에이전트를 만든다.

위임하면(test-analyst 에이전트로 테스트 돌려줘) 화면에 main과 test-analyst가 따로 뜨고, 장황한 빌드 로그는 에이전트 방에 남고 메인 대화엔 "통과 4 / 실패 0" 같은 요약만 온다. 이것이 컨텍스트 격리다(6장 그림 참고).

새 디렉토리 주의: .claude/agents/가 처음 생기는 경우, 실행 중이던 세션은 그 폴더를 못 보므로 재시작이 필요하다(자세한 규칙은 부록).

8. 오류 점검 & 실패 스토리

이 실습의 진짜 가치는 여기 있다. "됐다"가 아니라 "이렇게 막혔다가 이렇게 잡았다"의 기록이다.

fail-open — 안전장치가 실패하면 어디로 기우는가

delete-guard와 guard-error-codes 초기 버전은 JSON 파싱에 실패하면 catch { exit 0 }, 즉 그냥 통과(fail-open)시켰다. "검사 못 하면 통과"라 위험이 샐 수 있다. 반대는 "검사 못 하면 차단"인 fail-closed다.

  • 편의 훅(auto-fmt): fail-open이 낫다. 포맷 못 했다고 작업을 막으면 곤란하다.
  • 보안 훅(가드): fail-closed가 맞다. 검사 못 하면 일단 막는다.

matcher 구멍 — 실제 쓰인 도구를 커버해야 한다

delete-guard를 matcher: "Bash"로 걸었더니, rm -rf(Bash 도구)는 막혔지만 Remove-Item(PowerShell 도구)은 그대로 통과했다. 스크립트가 못 잡은 게 아니라, matcher에 안 걸려서 훅이 아예 호출되지 않은 것이다. matcher: "*"(모든 도구)로 넓혀 해결했다.

교훈: 단독 테스트는 통과인데 Claude Code에선 안 걸린다 → 십중팔구 matcher가 실제 쓰인 도구를 커버 못 한 것.

한글 인코딩 — Windows 훅의 함정

guard-error-codes가 9005 = force(영어)는 막는데 9005 = 강제 테스트(한글)는 통과시켰다. 원인: Windows PowerShell 5.1의 [Console]::In이 stdin을 UTF-8이 아니라 콘솔 코드페이지로 읽어, 한글이 든 JSON 파싱이 깨지고 → fail-open으로 새어나간 것이다. stdin을 UTF-8로 명시해 읽고(아래), fail-closed로 바꿔 해결했다.

$stdin = [Console]::OpenStandardInput()
$reader = New-Object IO.StreamReader($stdin, [Text.Encoding]::UTF8)
$raw = $reader.ReadToEnd()

Claude Code의 오진, 그리고 사용자의 정정

guard-error-codes가 안 걸린 원인을 두고 Claude Code는 추정을 오락가락했다: 처음 "인코딩 fail-open"(맞음) → 중간 "세션 스냅샷/미커밋 훅이라 로드 안 됨"(틀림) → 마지막 다시 인코딩(맞음). 실제로는 사용자가 스크립트를 고치고 다시 명령한 것이었고, 사용자가 "내가 방금 고쳤다"는 사실로 정정해 궤도를 잡았다.

교훈: Claude(Claude Code 포함)가 그럴듯하게 설명해도, 검증은 사람이 쥐고 있어야 한다. 단독 테스트·/hooks 확인·실제 결과 같은 사실이 추측을 이긴다.

9. 교훈 정리

  • 훅은 스크립트(무엇을 막나)와 matcher(어떤 도구에 거나)가 둘 다 맞아야 작동한다. 하나만 틀려도 샌다.
  • 보안 훅은 fail-closed, 편의 훅은 fail-open. 안전장치가 실패했을 때의 기본 동작이 곧 설계다.
  • 훅 테스트는 실제로 들어올 데이터로 한다. 영어로만 테스트하면 한글에서 새는 버그를 놓친다.
  • 인코딩은 Windows 훅의 단골 함정. PowerShell 5.1 stdin은 UTF-8 명시가 필요하다.
  • 스킬은 부탁, 훅은 강제. 진짜 지킬 규칙은 스킬로 안내하고 훅으로 강제한다(이중 방어).
  • 에이전트는 눈. 지저분한 작업을 딴 방에서 하고 결론만 회수해 메인 컨텍스트를 지킨다.
  • 검증은 사람이 쥔다. 훅을 만들었다고 끝이 아니라, 실제로 막히는지 스스로 확인해야 한다.

검증 명령어 모음 (cmd)

훅을 배선하기 전에 스크립트 단독으로 검증한다. 가짜 입력 JSON을 stdin으로 흘려 넣고 종료코드를 본다.

delete-guard — 위험 명령은 차단(exit 2), 안전 명령은 통과(exit 0):

type test-input.json | powershell -NoProfile -ExecutionPolicy Bypass -File .claude\hooks\delete-guard.ps1
echo exit=%errorlevel%

guard-error-codes — 9xxx(한글 포함)는 차단, 정상 코드는 통과:

echo {"tool_input":{"file_path":"src/main.rs","new_string":"// 9005 = 강제 테스트"}}> test-input.json
type test-input.json | powershell -NoProfile -ExecutionPolicy Bypass -File .claude\hooks\guard-error-codes.ps1
echo exit=%errorlevel%

echo {"tool_input":{"file_path":"src/main.rs","new_string":"// 2003 = 새 에러"}}> test-input.json
type test-input.json | powershell -NoProfile -ExecutionPolicy Bypass -File .claude\hooks\guard-error-codes.ps1
echo exit=%errorlevel%

auto-fmt — .rs면 fmt 실행, 아니면 통과(둘 다 exit 0):

echo {"tool_input":{"file_path":"src/main.rs"}}> test-input.json
type test-input.json | powershell -NoProfile -ExecutionPolicy Bypass -File .claude\hooks\auto-fmt.ps1
echo exit=%errorlevel%

기대값: 위험/9xxx는 exit=2, 안전/정상은 exit=0. 끝나면 del test-input.json.

파일 전문

src/main.rs

// 에러코드 체계
//   1xxx = 입력/파싱 오류
//   2xxx = 수학 오류
//   9xxx = 예약(사용 금지)
// 모든 에러는 stderr에 "[E<코드>] <메시지>" 형식으로 출력하고 종료한다.
//
// 사용 중인 코드:
//   1000 = 사용법 오류(인자 개수 불일치)
//   1001 = 숫자 아님
//   1002 = 지원하지 않는 연산자
//   2001 = 0으로 나누기
//   2002 = 곱셈 결과 오버플로우(무한대)
use std::process::exit;

/// 에러코드와 메시지를 "[E<코드>] <메시지>" 형식으로 stderr에 출력하고 종료한다.
fn die(code: u32, message: &str) -> ! {
    eprintln!("[E{}] {}", code, message);
    exit(1);
}

fn main() {
    let args: Vec<String> = std::env::args().collect();
    if args.len() != 4 {
        die(1000, "사용법: calc <숫자> <연산자> <숫자>   (예: calc 10 / 2)");
    }
    let a: f64 = args[1]
        .parse()
        .unwrap_or_else(|_| die(1001, &format!("숫자가 아닙니다: {}", args[1])));
    let op = &args[2];
    let b: f64 = args[3]
        .parse()
        .unwrap_or_else(|_| die(1001, &format!("숫자가 아닙니다: {}", args[3])));
    let result = match op.as_str() {
        "+" => a + b,
        "-" => a - b,
        "*" => {
            let m = a * b;
            if m.is_infinite() && a.is_finite() && b.is_finite() {
                die(2002, "곱셈 결과가 너무 커서 표현할 수 없습니다(오버플로우)");
            }
            m
        }
        "/" => {
            if b == 0.0 {
                die(2001, "0으로 나눌 수 없습니다");
            }
            a / b
        }
        _ => die(1002, &format!("지원하지 않는 연산자: {} (+, -, *, / 만 가능)", op)),
    };
    println!("{}", result);
}

.claude/hooks/delete-guard.ps1

$raw = [Console]::In.ReadToEnd()

$data = $null
try { $data = $raw | ConvertFrom-Json } catch { exit 0 }

$cmd = [string]$data.tool_input.command

$patterns = @('\brm\b', '\bdel\b', '\brd\b', '\brmdir\b', 'Remove-Item')
foreach ($p in $patterns) {
    if ($cmd -match $p) {
        [Console]::Error.WriteLine("[delete-guard] blocked delete command: $cmd")
        exit 2
    }
}
exit 0

.claude/hooks/guard-error-codes.ps1

# stdin을 UTF-8로 명시해서 읽기 (PowerShell 5.1 인코딩 문제 회피)
$stdin = [Console]::OpenStandardInput()
$reader = New-Object IO.StreamReader($stdin, [Text.Encoding]::UTF8)
$raw = $reader.ReadToEnd()

$path = ""
$newText = ""
try {
    $data = $raw | ConvertFrom-Json
    $path = [string]$data.tool_input.file_path
    $newText = [string]$data.tool_input.new_string + [string]$data.tool_input.content
} catch {
    # 파싱 실패 = 검사 불가 → 안전하게 차단 (fail-closed)
    [Console]::Error.WriteLine("[guard-error-codes] input parse failed - blocking to be safe")
    exit 2
}

if ($path -notmatch '\.rs$') { exit 0 }

if ($newText -match 'E9\d{3}' -or $newText -match '\b9\d{3}\s*=') {
    [Console]::Error.WriteLine("[guard-error-codes] 9xxx is a reserved range. Use 1xxx (input) or 2xxx (math) instead.")
    exit 2
}
exit 0

---
name: add-error-code
description: calc-harness 계산기에 새 에러코드를 추가할 때 사용. "에러 추가", "errorCode 추가", "새 에러 만들어" 등을 요청하면 반드시 이 스킬을 따른다.

# 에러코드 추가 규약

## 번호 범위 (엄수)
- 1xxx = 입력/파싱 오류
- 2xxx = 수학 오류
- 9xxx = 예약 — 절대 사용 금지

## 절차
1. 새 에러가 어느 범주인지 판단한다.
2. 해당 범위에서 이미 쓰인 번호를 확인하고, 다음 빈 번호를 고른다.
3. 9xxx는 어떤 경우에도 쓰지 않는다.
4. 출력 형식은 반드시 [E<코드>] <메시지> 로 맞춘다.
5. 에러는 stderr에 출력하고 exit(1)로 종료한다.

## 추가 후
- 추가한 코드와 의미를 파일 맨 위 주석에 기록한다.

재시작 규칙

공식 문서 기준으로, 무엇을 바꿨느냐에 따라 재시작 필요 여부가 다르다.

변경 재시작 근거(공식)
훅 스크립트(.ps1) 내용 불필요 훅은 매 호출마다 스크립트를 새로 실행
settings.json의 훅 등록/변경 원칙상 불필요 파일 워처가 변경을 자동 반영
기존 SKILL.md 내용 불필요 스킬 디렉토리를 워처가 감시
새 스킬 디렉토리(첫 생성) 필요 세션 시작 후 새로 생긴 최상위 스킬 폴더는 미감시
기존 에이전트 파일 추가/수정 불필요 agents 디렉토리를 워처가 감시
새 agents 디렉토리(첫 생성) 필요 세션 시작 시 없던 폴더는 미감시

실습 중에는 안전하게 매번 재시작했지만, 공식 문서에 따르면 settings.json 훅 변경은 파일 워처가 자동 반영한다. 확실히 하려면 재시작이 항상 안전한 방법이다. 새 스킬/에이전트 디렉토리를 처음 만든 경우는 실제로 재시작이 필요하다.

Windows/cmd 팁

  • .claude\settings.json 등 설정 파일은 BOM 없이 UTF-8로 저장한다. BOM이 있으면 첫 줄이 ---가 아니게 되어 스킬/에이전트 frontmatter를 못 읽는다. 확인: powershell -NoProfile -Command "Format-Hex -Path <파일> -Count 3" — 첫 3바이트가 EF BB BF면 BOM.
  • 순수 UTF-8(BOM 없음)로 쓰기: [System.IO.File]::WriteAllText("$PWD\.claude\settings.json", "{}")
  • 파일 내용의 숨은 문자(BOM/CR) 확인에 Format-Hex가 유용하다.
